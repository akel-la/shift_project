# ОПИСАНИЕ ПРОЕКТА:

Веб-сервис для автоматизации бронирования переговорных комнат в коворкинге.  

JWT-аутентификация.  
Валидация на входе - Pydantic  
Ендпоинты - FastAPI  
ORM - Асинхронная SQLAlchemy  
СУБД - Поддерживает SQLite (локально) и PostgreSQL (контейнер).  
Docker, Docker Compose - изоляция и переносимость.  


Тесты - и локально, и в контейнере, всегда используют SQLite:  
Фикстуры создают временную SQLite БД, миграции вообще не нужны.  

Разработка - локально, используем SQLite, миграции нужно применять вручную.  

Режим реальной работы - контейнер, PostgreSQL, миграции выполняются автоматически (скрипт entrypoint.sh).  

Используется общий интерфейс SQLAlchemy, нет специфической для той или иной СУБД логики, поэтому такое разделение можно считать нормальным.

Данные:

(по умолчанию) SQLite - данные в режиме реальной работы сохраняются в файле shift_data.db, в корне проекта.  

PostgreSQL - в docker compose файл указан как shift_postgres_data, поэтому в Linux он обычно хранится в:  
/var/lib/docker/volumes/shift_postgres_data/_data  
(стандартное место для хранения томов Docker).  

## ТРЕБОВАНИЯ:

- Python 3.12 или 3.13 (но не 3.14)
- Poetry (1.8+)
- Docker и Docker Compose (опционально, для запуска в контейнере)
- SQLite (локально) или PostgreSQL (контейнер)

Документация для работы с проектом в Linux.  
На Windows команды могут незначительно отличаться: используйте python вместо python3, del вместо rm, пути указывайте в кавычках.  


# ЗАПУСК ПРОЕКТА:

1. Клонируйте репозиторий:

```bash
git clone <url-репозитория>
cd shift_project
```

2. Настройте переменные окружения:

```bash
cp .env.example .env
```

3. Запустите скрипт, который создаст случайный секретный ключ:

```bash
python3 scripts/local/create_secret_key.py
```
затем скопируйте ключ в соответствующую переменную в .env.


4. Установите зависимости:

```bash
poetry install --with test --with dev
```

5. Примените миграции (вручную - только для локальной разработки):

```bash
poetry run alembic upgrade head
```

6. Запустить проект:

Локально:

```bash
poetry run uvicorn app.main:app --port 8000 --reload
```

В контейнере:

```bash
docker compose up -d --build
```

6. Взаимодействовать с проектом через GUI можно, открыв его в браузере по адресу:
http://localhost:8000/docs


# ОСТАНОВИТЬ ПРОЕКТ, ОЧИСТИТЬ ДАННЫЕ, НАЧАТЬ С НУЛЯ:

## ЛОКАЛЬНАЯ РАБОТА:

1. Закрыть терминал, в котором запущен uvicorn, или Ctrl + C.

2. Удалить файл shift_data.db в корне проекта.

3. Обновить alembic.

## РАБОТА В КОНТЕЙНЕРЕ:

1. Останавливаем контейнеры и очищаем volume:

```bash
docker compose down -v
```

2. Обновить alembic.

### ОБНОВИТЬ ALEMBIC:

1. Удалить все файлы миграций в alembic/versions/ (Не удалять \__init__.py).

2. Сгенерировать новую начальную миграцию:

```bash
poetry run alembic revision --autogenerate -m "initial"
```

3. Применить миграцию:

```bash
poetry run alembic upgrade head
```

# ЗАПУСК ТЕСТОВ:

Локально:

```bash
poetry run pytest
```

В контейнере (отдельный образ для тестов):

(Первый раз сборка займет много времени - до 5-8 минут)
```bash
docker build --target test -t shift_test .
docker run --rm --env-file .env shift_test
```

Если есть проблемы с тестами - можно в conftest поставить echo=True в движке.

# РАБОТА НАД ПРОЕКТОМ:

## СЕКРЕТНЫЙ КЛЮЧ:

Можно сгенерировать при помощи скрипта в ./scripts/local/create_secret_key.py, полученное значение записать в .env.

## alembic:

Если изменили слой классов-моделей ORM, то создаем и применяем миграцию, чтобы согласовать структуру таблиц в БД и ORM моделей в приложении:

```bash
poetry run alembic revision --autogenerate -m "<Вставьте описание - что изменили>"
poetry run alembic upgrade head
```

Проверить, что согласование произошло:

```bash
poetry run alembic current
```

Откатить последнюю миграцию:

```bash
poetry run alembic downgrade -1
```

## GIT:

Файл .env - должен быть в .gitignore, так как в .env находятся секретные данные.
Для того, чтобы хранить в проекте структуру, но не выдавать секретные данные - есть .env.example, в нем должны быть аналогичные переменные, но с другими значениями.

## БИБЛИОТЕКИ И POETRY:

Добавить библиотеку:

```bash
poetry add <имя>            # основная
poetry add --group test <имя> # тестовая
poetry add --group dev <имя>  # разработка
```

Обновить зависимости в poetry.lock:

```bash
poetry lock
```

Обновить pre-commit хуки:

```bash
poetry run pre-commit autoupdate
```

## ЗАМЕЧАНИЯ:


При запуске нескольких экземпляров приложения миграции выполняются каждым контейнером. Это безопасно, но может замедлить старт. В продакшене рекомендуется запускать миграции отдельно (например, через init-контейнер или CI/CD).


В .env (.env.example) - то, что общее для всех режимов работы (как локально, так и через Docker).
То, что нужно для Docker - находится в docker файлах.


scripts/runtime/entrypoint.sh - максимально простой.
