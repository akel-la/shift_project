# ОПИСАНИЕ ПРОЕКТА:

Веб-сервис для автоматизации бронирования переговорных комнат в коворкинге.

JWT-аутентификация.
Валидация на входе - Pydantic
Ендпоинты - FastAPI
ORM - Асинхронная SQLAlchemy
СУБД - Поддерживает SQLite (локально) и PostgreSQL (контейнер).
Docker, Docker Compose - изоляция и переносимость.


Тесты - и локально, и в контейнере, всегда используют SQLite:
Фикстуры создают временную SQLite БД, миграции вообще не нужны.

Разработка - локально, используем SQLite, миграции нужно применять вручную.

Режим реальной работы - контейнер, PostgreSQL, миграции выполняются автоматически (скрипт entrypoint.sh).

Используется общий интерфейс SQLAlchemy, нет специфической для той или иной СУБД логики, поэтому такое разделение можно считать нормальным.

## ТРЕБОВАНИЯ:

- Python 3.12 или 3.13 (но не 3.14)
- Poetry (1.8+)
- Docker и Docker Compose (опционально, для запуска в контейнере)
- SQLite (локально) или PostgreSQL (контейнер)

Документация для работы с проектом в Linux.
На Windows команды могут незначительно отличаться: используйте python вместо python3, del вместо rm, пути указывайте в кавычках.

# ПОЯСНЕНИЯ К ПРОЕКТУ:

## СТРУКТУРА ПРОЕКТА:

### Dockerfile:

base - это образ, на основе которого будут создаваться все конечные образы (test, prod), поэтому base это slim образ (ничего лишнего) и в нем все то, что является общим для всех конечных образов.

test и prod - это конечные образы.

builder образы - они для сборки (компиляция, и тому подобное), поэтому они отдельно и они НЕ slim. Мы собираем в них все необходимое, а затем просто копируем это через COPY в конечные образы.

Флаг --no-root - не устанавливать сам проект, общая логика в том, чтобы разделить слои для кеширования:

1. Файлы pyproject.toml и poetry.lock:
COPY pyproject.toml poetry.lock ./

2. Зависимости, являющиеся ядром проекта:
RUN poetry install --no-dev --no-root

3. Код проекта:
COPY ./src ./src

4. Установка самого проекта:
RUN poetry install --no-dev

### Alembic:

Не нужно указывать DATABASE_URL в 2 местах - alembic.ini и env.py, достаточно только в env.py.

То, что общее для всего проекта:
```sh
/core:
   # Класс Base(DeclarativeBase), от которого наследуют все классы-таблицы:
   base.py
   # Настройки:
   config.py
   #
```

### SQLAlchemy:

У UniqueCOnstraint есть свои имена, соответствующие шаблону, в котором перечисленны все столбцы, которые участвуют в проверке:

uq_<столбец_1>_<столбец_2>...

### КАТАЛОГИ И УРОВНИ:

Уровень репозиториев:
    Не делают коммитов.
    Не работают с Pydantic схемами.
    Возвращают экземпляры Классов-таблиц.
    Не проверяют на ошибки.

1. src - там только код самого проекта.
   src/app - общая папка самого проекта.
2. tests - тесты к проекту.
3. Родительский каталог - вспомогательное (git), оболочки (Docker, .venv) и прочее подобное.

В pyproject.toml:
```toml
packages = [{include = "app", from = "src"}]
```
Позволяет писать простой импорт:

from app import ...

обращаясь к приложению как к пакету.

src - это просто папка, это не пакет - поэтому в src нет \__init__.py.
app - это пакет, поэтому внутри \__init__.py.
