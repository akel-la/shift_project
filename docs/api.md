# MODEL_DUMP:

У нас есть схема для POST (создание) запросов, у нее есть поля со значениями по умолчанию. Для обновления мы используем эту же схему. Если пользователь прислал новые значения для одних полей, то значения по умолчанию для других полей могут изменить данные, которые не должны были измениться.
```python
room = await service.update(room_id, **data.model_dump(exclude_unset=True))
```

model_dump возвращает словарь из объекта, благодаря exclude_unset=True в этот словарь попадают только поля, которые явно заданы при запросе (поля со значениями по умолчанию отсутствуют). Смысл - запрос обновляет только те поля, которые были явно присланы (PATCH запрос), и не затирает поля, которые запрос не планирует менять (PUT).

# USER, ADMIN И AUTH:

auth - регистрация сотрудников (публичная), аутентификация, работа с токенами.
admin - создание администраторов.
user - управление уже существующими пользователями.

# URL:

Если сущностью можно управлять отдельно, то не нужно делать сложные URL, с упоминанием других сущностей. Мы делаем URL более простыми и скрываем от пользователя структуру бекенда, пользователям не нужно об этом знать и мы можем менять структуру не меняя ендпоинты.

Пример:
Слоты связанны с комнатами. Можно ли работать со слотами, имея только их id (slot_id), но не имея id комнат? Если да, то URL слотов не включает в себя "/rooms" или "/rooms/{room_id}".

Если есть отдельные операции над слотами, которые требуют room_id, то только тогда добавляем room_id в URL:
"rooms/{room_id}/slots".

# LOAD:

На уровне API везде по умолчанию = False, чтобы при поведении по умолчанию было меньше запросов в БД.

# PATCH И PUT:

PATCH - можно прислать не все поля, метод не идемпотентный.
PUT - нужно прислать все поля, метод идемпотентный.

В проекте методы update - можно прислать не все поля (как в PATCH), идемпотентные методы (как в PUT). Решил, что подчеркнуть идемпотентность важнее, поэтому PUT.

# CreateResponse:

И в create, и в update в качестве ответа будут схемы CreateResponse - с одной стороны, не добавляем лишних схем (CreateResponse, UpdateResponse, ...), а с другой стороны заранее делаем разделение между Response и CreateResponse - как минимум, у только что созданного объекта точно не будет связанных объектов (если нет ендпоинтов, которые позволяют, к примеру, сразу создать комнату и ее слоты). Даже если сейчас CreateResponse и Response одинаковые, в будущем это может быть не так.

# ПРОВЕРКА ПРАВ:

Если требуются права админа, то проверяем это через зависимость - get_current_admin_user:
```python
@router.post("/rooms", response_model=RoomResponse, status_code=status.HTTP_201_CREATED)
async def create_room(
    session: Annotated[AsyncSession, Depends(get_async_session)],
    _admin: Annotated[User, Depends(get_current_admin_user)],
    data: RoomCreate,
    ):
```
Если с символа нижнего подчеркивания - значит, просто проверка, без передачи объекта-пользователя на уровень бизнес логики. В противном случае вместо _admin будет current_user.

# ПЕРЕХОД НА PYTHON 3.14:

Требует переделки проекта, так как в 3.14 поменялась скрытая механика аннотаций.
Код вида:
````python
_admin: Annotated[User, Depends(get_current_admin_user)]
```
Будет вызывать ошибки.
Если вместо User поставить Any, то ошибки не будет (так же есть и другие способы). Причина - особенности работы отложенных аннотаций и тонкости взаимодействия FastAPI и SQLAlchemy, когда класс SQLAlchemy является аннотацией в Annotated[..., Depends(...)].
Подробнее:
https://errors.pydantic.dev/2.13/u/class-not-fully-defined

Решения:
1. Отказаться от Annotated и вместо этого использовать синтаксис:

_admin: User = Depends(get_current_admin_user)

2. Использовать в аннотации Any:

_admin: Annotated[Any, Depends(get_current_admin_user)]

Если нужно, чтобы работала аннотация, то можно в теле эндпоинтов написать:

_amin_user: User = _admin
