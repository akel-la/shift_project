# Dockerfile:

## ОБРАЗЫ И ИХ ВЗАИМОСВЯЗЬ:

base - это образ, на основе которого будут создаваться все конечные образы (test, prod), поэтому base это slim образ (ничего лишнего) и в нем все то, что является общим для всех конечных образов.

test и prod - это конечные образы.

builder образы - они для сборки (компиляция, и тому подобное), поэтому они отдельно и они НЕ slim. Мы собираем в них все необходимое, а затем просто копируем это через COPY в конечные образы.

Флаг --no-root - не устанавливать сам проект, общая логика в том, чтобы разделить слои для кеширования:

# СЛОИ - ПОРЯДОК И КЭШИРОВАНИЕ:

1. Файлы pyproject.toml и poetry.lock:
COPY pyproject.toml poetry.lock ./

2. Зависимости, являющиеся ядром проекта:
RUN poetry install --no-dev --no-root

3. Код проекта:
COPY ./src ./src

4. Установка самого проекта:
RUN poetry install --no-dev

# ROOT И БЕЗОПАСНОСТЬ:

По умолчанию Docker работает от лица root, что опасно.
Здесь:

```Dockerfile
RUN groupadd -r -g 999 app && \
     useradd -r -u 999 -g app -s /usr/sbin/nologin app
```

Создаем группу пользователей под названием app,

флаг: -r указывает, что созданная группа должна быть системной группой.
Обычные группы, ID >= 1000 - создаются для реальных людей.
Системные группы, ID <= 999 - создаются самой ОС или приложениями, нужны для работы фоновых процессов. В Linux разница чисто формальная, различие только по диапазонам ID, в других ОС по-другому.

Аналогично, с обычными и системными пользователями.
Системные пользователи - не могут удаленно входить в систему, у них вместо оболочки /bin/false или /bin/nologin (но на всякий случай вручную зададим nologin), не имеют пароля.

useradd ... app - создаем пользователя app, флаги:

Создание системного пользователя: -r

Указать, что для пользователя основная группа это группа app: -g app

Указывает язык-оболочку /usr/sbin/nologin для пользователя: -s /usr/sbin/nologin

Мы создаем именно группу, а не просто пользователя, потому, что:

(1) Если создать просто пользователя - система может автоматически назначить ему общую группу, с лишними правами. Также, если пользователь системный, то и группа у него должна быть системной.

(2) Если с проектом взаимодействуют несколько процессов - к примеру, само приложение, процесс мониторинга и тому подобное, то проще объединить их в одну группу с общими правами.

(3) Совместимость - многие Linux инструменты и скрипты ожидают, что у приложений есть своя одноименная группа.

# ПРИ ИЗМЕНЕНИИ ПРОЕКТА:

При запуске нескольких экземпляров приложения миграции выполняются каждым контейнером. Это безопасно, но может замедлить старт. В продакшене рекомендуется запускать миграции отдельно (например, через init-контейнер или CI/CD).
