# ОБЩАЯ АРХИТЕКТУРА:

Не возбуждают исключений.

Не содержат проверку прав/полномочий.

Не делают session.commit() - это должны делать сервисы, один commit на одну бизнес операцию, но делают session.flush() - чтобы БД сделала проверки (уникальность и прочее) и чтобы получить данные (к примеру, id), которые могут понадобиться бизнес логике.

Если то, что запрашивают, не существует - возвращают None. Проверки на None - во вспомогательных методах сервисов.

Коллекция на входе - Sequence, для гибкости.
Коллекция на выходе - list, для единообразия, и аннотируем как Sequence, чтобы указать, что по объекту можно итерироваться, но без подробностей этого объекта.

Чтобы было проще искать - операции идут в CRUD порядке: create, read (get), update, delete, а так же от более простых и обычных к более сложным (get_by_id, get_by_name, get_by_datetime_filter и.т.д.).

# Параметры load_*:

Для загрузки связанных данных:

(+) Меньше запросов к БД - нет Query N+1 проблемы на уровне кода бизнес логики (бизнес логике достаточно сделать только один запрос к БД через репозиторий).

Вложенность - только один уровень (комната.слоты, но не комната.слоты.бронирования), иначе слишком большие объемы данных, сложный код, смещение ответственности.

Пример:

Комната и все ее слоты - один запрос (load_slot = True)
Комната, все ее слоты и все бронирования этих слотов - 2 запроса: "комната и все слоты", "все бронирования комнаты".
