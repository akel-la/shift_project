# ОБЩАЯ АРХИТЕКТУРА:

Если действие может выполнить только админ - то проверка на админа на уровне ендпоинтов.
Если действие может выполнить и обычный пользователь - то там более сложная логика (какой пользователь с какими полномочиями что может делать с с собой и остальными), проверка прав на этом уровне.

Одна бизнес-операция - один session.commit().

# АСИНХРОННЫЙ DELETE:

На уровне сервисов delete асинхронный, поэтому нужен await:
```python
await self.repo.delete(room)
```
как минимум, так как внутри метода delete на уровне репозитория есть
```python
await self.session.flush()
```

# СВЯЗЬ СЕРВИСОВ:

Для простых случаев - каждый класс-сервис обращается только к соответствующему классу-репозиторию (RoomService --> RoomRepository).

Сервис может обращаться к нескольким разным классам-репозиториям.

Сервис НЕ ДОЛЖЕН обращаться к другим сервисам, так как это может привести к циклической зависимости, пример:

```python
class RoomService:
    def __init__(self, session: AsyncSession):
        ...
        self.slot_service = SlotService(session)

class SlotService:
    def __init__(self, session: AsyncSession):
        ...
        self.room_service = RoomService(session)
```
Код не может работать, так как циклическая зависимость.

В итоге приходится иногда немного дублировать функционал, пример:

```python
class SlotService:
    ...

    async def create(...):
        ...
        room = await self.room_repo.get_by_id(room_id)
        if not room:
            raise NotFoundError(f"Комната с d = {room_id} не найдена.")
        ...
```
Идет обращение к RoomRepository, а затем проверка, хотя такой фрагмент кода есть в RoomService.
