## CREATE_INITIAL_ADMIN:

У app = FastAPI(...) есть lifespan, который создает админа в БД, если переменная CREATE_INITIAL_ADMIN = true.
По умолчанию установлено true.
При тестировании мы работаем с тестовой БД, все данные в ней создаются либо фикстурами, либо тестами, к тому же, нам не нужно, чтобы при работе тестов была попытка добавить админа в реальную БД (происходят оба сценария?).
Чтобы этого не было - переопределяем переменную окружения ПЕРЕД импортом app = FastAPI(...).

```python
import os # noqa
os.environ["CREATE_INITIAL_ADMIN"] = "false"   # noqa
```

## APP:

Есть объект app из main.py:

```python
app = FastAPI(...)
```

Есть пакет app.

В случае, когда нужно импортировать сам пакет (к примеру, чтобы отработал код внутри его \__init__.py), то может возникнуть путаница, если пакет и объект FastAPI названы одинаково.

Решение:
```python
from app.main import app as application
```

# БД:

SQLite - БД в оперативной памяти была бы быстрее, чем временный файл в массовой памяти, но такая БД работает в рамках одного соединения. В ситуации, когда у разных тестов и фикстур свои соединения, такой вариант не подходит (одно соединение - создание таблиц, другое соединение - работа с этими таблицами). Вместо этого - временный файл, который видят все.

## ФИКСТУРЫ db_session И clean_db:

### db_session:

Сложное устройство:

Внешняя транзакция - для самих тестов, суть в том, чтобы отменить то, что сделал тест.
Внешняя транзакция никогда не commit, она только rollback.

Внутренняя транзакция - передаем коду, который тестируем, чтобы он делал с ней все, что нужно:
commit (позитивное тестирование),
rollback (негативное тестирование),
save point,
и.т.д.

Код:

```python
async with async_session_factory() as session:
    # outer_transaction - внешняя транзакция.
    # Не используем with ... as ..., так как в этой ситуации контекстный менеджер делает автоматический commit:
    outer_transaction = await session.begin()
    try:
        # Вложенная транзакция (для тестируемого кода):
        await session.begin_nested()
        yield session
    # При любом сценарии, в конце ...
    finally:
        # пытаемся rollback вложенную транзакцию:
        try:
            await outer_transaction.rollback()
        # Если вложенная транзакция уже была rollback тестируемым кодом ...
        except ResourceClosedError:
            # То просто перехватываем исключение и все:
            pass
```
Если нужно дополнительно проверить, что, к примеру, не только HTTP ответ говорит "не получилось добавить данные в БД", но и сам сервис реально не добавил данные в БД - то просто в коде теста делаем GET запрос и ожидаем 404.

Так как внутри тестируемого кода (разные методы разных сервисов) сложная логика, которая по-разному работает (в зависимости от поступающих данных, и того, позитивное или негативное тестирование), то даже сложное устройство db_session не дает гарантии того, что БД вернется в то состояние, в котором она была до тестов.
Решение - дополнительная фикстура clean_db.

### clean_db:

Очищает все таблицы после каждого теста.

(+) Простая логика - нет внешних и внутренних транзакций,
перехвата (try / except) ResourceClosedError и прочего подобного.

(+) Гарантия чистоты БД, вне зависимости от того,
какие commit / rollback и прочие сложные манипуляции
делает тестируемый код.

(+) Быстрее, чем после каждого теста удалять таблицы
(или саму БД), а затем перед следующим тестом заново их создавать.

(-) Не позволяет один раз добавить данные в БД для всех тестов,
но такое поведение обычно не делается, чтобы гарантировать изоляцию.
